# specPPT

一个专门用于 **PPT 内容设计** 的强引导型 skill。

它只负责 **内容设计**，不负责视觉执行、不负责直接出 PPT、不负责 SVG 或 PPTX 生成。

它的目标是把用户的需求、边界、受众、核心判断和页面解释任务，沉淀成 **可直接交给 PPT 执行类 skill 使用的 markdown 文件**，并尽量避免后续制作过程再次漂移、浅化或另起一套叙述。

---

## 一句话定位

**先把 PPT 要讲清什么、怎么论证、每页负责解释什么锁定，再进入制作。**

---

## 何时使用

当用户有以下需求时，优先使用本 skill：

- 从零开始设计一份 PPT / 汇报 / 分享材料
- 已经有初稿、大纲或成品，但觉得内容浅、结构弱、逻辑散，需要回退重构
- 想先做内容设计，再交给别的技能或工具制作 PPT
- 不希望一上来就写“标题 + bullets”，而是先把论证结构理顺
- 需要把复杂方法论翻译成更适合业务受众的表达

典型触发语包括但不限于：

- “先别画 PPT，先把内容结构做出来”
- “帮我写一版 spec / 页面级大纲”
- “这个大纲太浅了，回退重做内容设计”
- “我要做一份汇报，先帮我把内容论证结构想清楚”
- “把这份已有 PPT 回退到内容设计层重构”

---

## 不做什么

本 skill 默认 **不做** 以下事情：

- 不直接生成 PPT / SVG / PPTX
- 不直接进入视觉排版
- 不代替用户选择模板、颜色、版式细节
- 不把内容过早压缩成只剩标题和 bullet 的页面卡片
- 不把讲稿写成独立于页面结构之外的另一套叙述

如果后续要制作 PPT，应把本 skill 的输出交给专门的执行型技能，例如 `ppt-master`。

---

## 默认输出

本 skill 默认输出以下 3 个 markdown 文件：

1. `spec.md`
2. `page_outline.md`
3. `speaker_notes.md`

如当前阶段只做到内容设计，也可以先完成前两个文件，再视需要补 `speaker_notes.md`。

不默认单独输出 `tasks.md`。

如需交接执行，可在 `page_outline.md` 末尾附一个轻量 handoff checklist。

## 双模式

### 模式 A：`new`
用于 **从零开始** 设计一份 PPT / 汇报内容。

适用场景：
- 用户只有主题、目的、受众、零散想法或参考材料
- 还没有成型大纲
- 希望从需求澄清一路推进到页面级内容设计

#### `new` 模式开场模板
当用户是从零开始时，应优先用一个 **结构化开场 intake** 收束信息，而不是直接开始分页。

推荐优先收集以下字段：
- 主题 / 暂定题目
- 汇报对象
- 汇报场景
- 汇报目标
- 这次明确不讲什么
- 用户已有粗想法
- 是否已有参考材料
- 页数范围
- 风格要求
- 用户最想让听众带走的 1–3 个判断

如果用户信息不完整，可用如下模板推进：

```md
先锁定这份汇报的内容设计输入，我建议先按这 10 项收束：

1. 主题 / 暂定题目：
2. 汇报对象：
3. 汇报场景：
4. 汇报目标：
5. 明确不讲什么：
6. 你已经有的粗想法：
7. 现有参考材料：
8. 预计页数范围：
9. 风格要求 / 禁区：
10. 听众最终应带走的 1–3 个核心判断：
```

如果用户只给了零散想法，也应主动整理成以下归纳块再确认：

```md
### 当前已知
- 主题：...
- 受众：...
- 目标：...
- 非目标：...
- 风格：...
- 已有素材：...

### 目前缺口
- ...
- ...

### 我建议先锁定
- ...
- ...
```

### 模式 B：`rebuild`
用于 **已有半成品，需要回退到内容设计层重构**。

适用场景：
- 已有 spec、大纲、讲稿或 PPT 成品
- 用户认为内容太浅、逻辑太散、页面职责不清
- 需要先诊断问题出在哪一层，再重建结构

#### `rebuild` 模式开场模板
当用户要重构已有内容时，不能只问“哪里不对”，必须先把 **现状、症状、保留项、重做范围、根因层判断** 分开。

推荐优先收集以下字段：
- 当前已有材料形态（spec / 大纲 / 讲稿 / 成品 PPT）
- 用户感受到的表层问题
- 用户判断最严重的 1–3 个问题
- 希望保留的部分
- 希望重做的部分
- 当前目标受众和使用场景
- 原先想讲但被漏掉的点
- 用户怀疑问题出在哪一层

推荐使用如下 intake 模板：

```md
先把这次 rebuild 的边界锁定，我建议按这 8 项诊断：

1. 当前已有材料是什么：
2. 你现在最不满意的表层症状是什么：
3. 你认为最严重的 1–3 个问题：
4. 哪些内容你希望保留：
5. 哪些内容你希望重做：
6. 当前真实受众和使用场景：
7. 原先想讲但这一版漏掉了什么：
8. 你怀疑问题更像出在 Proposal / Requirements / Argument Design / Page Outline 哪一层：
```

在用户给出材料后，应先整理成诊断块：

```md
### 当前版本的主要症状
- ...
- ...

### 建议保留
- ...
- ...

### 建议重做
- ...
- ...

### 初步判断问题层级
- Proposal：...
- Requirements：...
- Argument Design：...
- Page Outline：...
```

## 核心原则

### 1. 先论证，后压缩
先锁定：
- 主问题
- 汇报目标
- 受众边界
- 核心判断
- 论证主线
- 每页解释任务

再压缩成适合 PPT 的表达。

### 2. 页面不是结论卡片，而是解释单元
每页都必须先回答：
- 这页到底在解释什么
- 为什么这页必须存在
- 它和前后页怎么连接

### 3. 图示必须承担解释责任
图示不能只是把一句结论再画一遍。
图示必须明确负责：
- 因果解释
- 结构拆解
- 对比澄清
- 过程展示
- 层次递进
中的至少一种。

### 4. 先适配受众，再选择表达密度
表达必须服务于受众，不默认炫技，不默认技术布道。

### 5. 不把用户变成逐字撰稿人
用户负责提供方向、边界、主旨、纠偏和确认。
本 skill 负责把这些信息发展成完整、可执行的内容结构。

## 工作流总览

本 skill 采用强引导、严格分阶段推进。

### Stage 1 — Proposal
锁定这份 PPT 的任务定义。

需要明确：
- 这份汇报要解决什么问题
- 汇报对象是谁
- 这次不讲什么
- 用户最想让听众带走什么

**本阶段输出应至少形成：**
- 汇报主题
- 汇报目标
- 非目标
- 受众画像
- 使用场景

---

### Stage 2 — Requirements
把需求收束成清晰边界。

重点明确：
- 风格要求
- 表达禁区
- 内容边界
- 是否需要例子
- 是否要偏业务翻译
- 是否允许技术术语
- 页数范围
- 当前阶段产物范围

**本阶段结束前必须有第一次确认。**

### Gate 1：Requirements Freeze
在进入论证设计前，必须请用户确认或修正：
- 主题
- 目标
- 受众
- 风格
- 页数范围
- 明确不讲什么

未确认前，不得直接展开完整页面设计。

---

### Stage 3 — Argument Design
这是本 skill 的核心阶段。

要完成的不是“列提纲”，而是建立整场内容的论证骨架。

至少要形成：
- 主问题
- 2–4 条主线
- 3–6 个核心判断
- 各部分之间的推进关系
- 哪些页负责解释机制
- 哪些页负责纠偏误解
- 哪些页负责方法迁移
- 哪些页负责业务落地

对于 `rebuild` 模式，必须先做一段诊断：

### Rebuild Diagnostic
先判断现有问题主要出在哪一层：
- Proposal 层：一开始要解决的问题就不清楚
- Requirements 层：受众、边界、风格没有锁住
- Argument Design 层：主线、核心判断、推进关系不稳
- Page Outline 层：页面被过早压缩，解释任务丢失

诊断后，再进入重建设计。

**本阶段结束前必须有第二次确认。**

### Gate 2：Argument Design Freeze
在进入页面设计前，必须请用户确认或修正：
- 主问题
- 主线结构
- 核心判断
- 大致章节划分
- 是否存在重要遗漏

未确认前，不得直接产出完整页面级大纲。

---

### Stage 4 — Page Outline
把论证骨架展开成页面级内容设计。

这一阶段不能只写标题和 bullet。

默认每页采用以下结构：

- 页面标题
- 页面问题
- 核心判断
- 推理链
- 正文承载信息
- 图示责任
- 例子 / 辅助表达
- 页面落点句

必要时可扩展：
- 业务翻译
- 易误解点
- 与上一页 / 下一页的衔接作用
- 必须保留表述
- 明确禁区

#### 页面级设计要求

每页都要明确：
1. **页面问题**：这页回答什么问题
2. **核心判断**：这页最重要的一句判断
3. **推理链**：为什么得出这个判断，至少写出 3 个推进节点；不能只剩结论复述
4. **正文承载信息**：这页正文区域必须出现哪些信息块、它们的先后顺序是什么、哪些信息必须落在正文而不能只留给讲稿或执行时自由发挥
5. **图示责任**：图要解释什么，而不是准备放什么图；至少写清图的结构骨架、组成模块、模块关系、图中必须出现的标签或对照项
6. **例子 / 辅助表达**：帮助理解，不是装饰；例子后需补一句业务翻译或理解落点
7. **页面落点句**：本页收束到哪里

如果某页需要图示，默认至少写清以下 4 项：
- 图的结构骨架（如三栏对比 / 两段流程 / 四层阶梯 / 中心-外环）
- 图的组成模块（每个区块分别承载什么信息）
- 模块关系（并列 / 递进 / 因果 / 对照 / 闭环）
- 图中必须出现的关键词或标签

如果某页未来要交给执行型 PPT skill，内容设计层不得只写“放一个流程图/示意图/对比图”就结束；必须把图的解释负担拆到可执行程度。

如果某页在执行时容易被过度压缩，需额外补：
- **必须保留表述**：哪些词句或判断不能被执行时改写掉
- **明确禁区**：哪些误读、俗套表达或错误改写不能出现

**本阶段结束前必须有第三次确认。**

### Gate 3：Page Outline Freeze
在进入讲稿或交接前，必须请用户确认或修正：
- 页面顺序是否合理
- 每页是否讲透
- 是否有明显过浅或重复页面
- 图示责任是否足够清楚
- 标题风格是否统一

未确认前，不得跳到执行制作。

---

### Stage 5 — Speaker Notes
在页面结构冻结后，再写讲稿或讲述辅助内容。

要求：
- 讲稿必须服务于页面逻辑
- 不得另起一套叙述
- 优先补足口头过渡、例子展开、解释顺序
- 不把讲稿写成与 PPT 无关的文章

如果用户当前只需要内容设计，可以先停在 Stage 4。

---

### Stage 6 — Handoff
如果要交给执行型 PPT skill，需整理最小交接信息。

建议在 `page_outline.md` 文末附：
- 总页数
- 各页标题
- 每页图示责任
- 是否有重点示意图页
- 是否有不宜再压缩的页面
- 是否有必须保留的例子或原句

## 用户通常需要提供什么

用户不需要逐字逐句写完整内容。

通常只需要尽量提供以下信息中的一部分：

### 在 `new` 模式下
- 主题 / 题目
- 汇报对象
- 汇报场景
- 目标
- 自己已有的粗想法
- 不想讲什么
- 参考材料
- 风格偏好

### 在 `rebuild` 模式下
- 现有 spec / 大纲 / 讲稿 / PPT
- 自己认为哪里不对
- 希望保留什么
- 希望重做什么
- 目标受众和使用场景

如果用户提供的信息不完整，本 skill 应通过分阶段澄清来补齐，而不是要求用户一开始一次性写完全部内容。

## 默认风格偏好

以下规则可作为默认偏好，除非用户明确覆盖：

1. **先保留论证结构，再压缩成 PPT 文案**
2. **标题优先准确清晰，不强行口号化**
3. **总结页用克制、务实、业务化表达，不要太 AI 味**
4. **例子要短、准、直接服务理解**
5. **图示必须承担解释责任，不能空转**
6. **优先把方法论翻译成业务语言，而不是技术术语堆砌**

---

## 高风险反模式

### 1. 过早压缩
错误做法：一开始就写成“标题 + bullet + 例子 + 收束句”。

风险：
- 页面会浅
- 论证关系会断
- 后续制作时容易自由发挥

### 2. 图示空转
错误做法：只说“这里放个流程图 / 示意图”。

正确做法：明确图示到底负责解释什么。

### 3. 只有结论，没有推理负担
错误做法：每页只有观点，没有说明它为何成立。

### 4. 讲稿先行
错误做法：页面结构还没定，就先写长段讲稿。

### 5. 术语感太重，受众适配不足
错误做法：默认用技术圈表达，不做业务翻译。

### 6. 应用部分与前文脱节
错误做法：前面讲原理，后面突然罗列应用清单。

正确做法：应用页必须承接前文方法与边界。

## 执行纪律

1. 必须严格按阶段推进，不跨阶段偷跑。
2. 在 Gate 1、Gate 2、Gate 3 处必须征求用户确认。
3. 若用户直接要求“先写 PPT 文案”，也要先判断是否缺失前置内容设计。
4. 若用户已有成品且问题明显，应优先建议使用 `rebuild` 模式。
5. 若用户内容很散，应优先先收束 Proposal 和 Requirements，而不是急着分页。
6. 任何时候都要防止把 rich content logic 压缩成浅层页面卡片。

---

## 与执行型 PPT 技能的关系

本 skill 与 `ppt-master` 的分工如下：

- `specPPT`：负责内容设计
- `ppt-master`：负责视觉执行与导出

推荐顺序：

1. 用 `specPPT` 产出 `spec.md`
2. 用 `specPPT` 产出 `page_outline.md`
3. 如需要，再由 `specPPT` 补 `speaker_notes.md`
4. 再把这些 markdown 交给 `ppt-master` 制作 PPT

## 最简使用方式

### 用法 1：从零开始
用户可以这样说：

- “用 specPPT 帮我从零设计一份关于 XX 的汇报内容。”
- “先不要做 PPT，先用 specPPT 帮我把内容结构做出来。”

### 用法 2：回退重构
用户可以这样说：

- “用 specPPT 的 rebuild 模式，帮我重构这份已有大纲。”
- “这版 PPT 太浅了，回退到内容设计层重做。”

---

## 结束标准

本 skill 的合格结束状态不是“有几页标题”，而是：

- 主问题清楚
- 主线清楚
- 核心判断清楚
- 页面职责清楚
- 图示责任清楚
- 后续执行不会轻易漂移

如果这些还没有清楚，就不应视为完成。
